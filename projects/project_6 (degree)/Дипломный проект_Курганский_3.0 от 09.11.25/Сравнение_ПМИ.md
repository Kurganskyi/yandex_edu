# Сравнение ПМИ: пример vs созданный документ

## Основные различия

### 1. Структура основных разделов

**Пример (pmi_stets_ex.md):**
- Использует одну таблицу для всех основных разделов:
  - Объект испытаний
  - Цель испытаний
  - Требования к программе
  - Требования к документации
  - Средства и порядок испытаний
- Все в одной таблице с двумя столбцами

**Созданный документ (ПМИ_Stets_Home.md):**
- Использует отдельные заголовки для каждого раздела
- Каждый раздел оформлен как отдельная секция с подзаголовками
- Более детальное описание в разделе "Средства и порядок испытаний"

**Вывод:** Пример более компактный, созданный документ более структурированный и детальный.

---

### 2. Название раздела методики

**Пример:**
- "Методы испытаний"

**Созданный документ:**
- "Методика испытаний"

**Вывод:** Разные названия, но по сути одно и то же.

---

### 3. Структура таблицы методики испытаний

**Пример:**
- 4 столбца:
  1. Требование технического задания
  2. Предусловия
  3. Шаг проверки
  4. Конечный результат

**Созданный документ:**
- 5 столбцов:
  1. Требование
  2. Предусловия
  3. Шаг проверки
  4. Ожидаемый результат
  5. Связанные Gherkin-сценарии

**Вывод:** Созданный документ добавляет столбец с Gherkin-сценариями для связи с пользовательскими историями.

---

### 4. Идентификация требований

**Пример:**
- Использует технические требования: **TR-1**, **TR-1.1**, **TR-1.2**, **TR-2**, **TR-2.1** и т.д.
- Структура: TR-X (основное требование), TR-X.Y (подтребования)

**Созданный документ:**
- Использует пользовательские истории: **US-001**, **US-002**, **US-003** и т.д.
- Структура: US-XXX (номер пользовательской истории)
- Для вариаций сценариев: "US-001: Регистрация - невалидный email"

**Вывод:** Разные системы идентификации - пример использует технические требования, созданный документ использует пользовательские истории из карты.

---

### 5. Детализация шагов проверки

**Пример:**
- Более краткие описания
- Меньше примеров в шагах
- Более общие формулировки

**Пример шага:**
```
1. На экране входа нажать «Зарегистрироваться».
2. Указать валидные значения в полях «Имя» и «Электронная почта».
3. Ввести и повторить пароль (соответствует требованиям).
4. Нажать «Создать аккаунт».
```

**Созданный документ:**
- Более детальные описания
- Конкретные примеры значений (например, "mikhail@email.com", "MyPass123")
- Более подробные формулировки

**Пример шага:**
```
1. Открыть экран регистрации (нажать "Зарегистрироваться" на экране входа).
2. Заполнить поле "Имя" валидным значением (например, "Михаил").
3. Заполнить поле "Email" валидным email (например, "mikhail@email.com").
4. Заполнить поле "Пароль" паролем, соответствующим требованиям (8-16 символов, строчные и прописные буквы, например "MyPass123").
5. Заполнить поле "Повторите пароль" тем же паролем.
6. Нажать кнопку "Создать аккаунт".
```

**Вывод:** Созданный документ более детальный и содержит конкретные примеры.

---

### 6. Описание ожидаемых результатов

**Пример:**
- Более краткие формулировки
- Фокус на ключевых результатах

**Пример:**
```
Кнопка становится активной. Пользователь успешно зарегистрирован. На email приходит письмо о регистрации.
```

**Созданный документ:**
- Более развернутые описания
- Включает детали о переходе на другие экраны
- Упоминает связанные функции (например, автосоздание дома)

**Пример:**
```
Кнопка становится активной при заполнении всех полей. После нажатия пользователь успешно зарегистрирован. На главном экране отображается дом "Мой дом". Пользователь попадает на главный экран (Dashboard).
```

**Вывод:** Созданный документ более подробно описывает ожидаемые результаты.

---

### 7. Раздел "Средства и порядок испытаний"

**Пример:**
- Краткое описание в одной строке таблицы:
  ```
  • Мобильный телефон с iOS/Android
  • Доступ в интернет
  ```

**Созданный документ:**
- Детальный раздел с подразделами:
  - Средства испытаний (прототип, оборудование, документация)
  - Порядок испытаний (подготовка, проведение, документирование)
- Более структурированное описание

**Вывод:** Созданный документ содержит более подробное описание средств и порядка испытаний.

---

### 8. Раздел "Требования к программной документации"

**Пример:**
- Краткое упоминание в таблице:
  ```
  Программа и методика испытаний (ПМИ) сервиса
  ```

**Созданный документ:**
- Отдельный раздел со списком всех необходимых документов:
  1. ПМИ
  2. Карта пользовательских историй
  3. DFD диаграммы
  4. ER-диаграмма
  5. Словарь данных
  6. Диаграмма структуры интерфейса
  7. Список экранов прототипа

**Вывод:** Созданный документ более детально описывает необходимую документацию.

---

### 9. Раздел "Требования к программе"

**Пример:**
- Перечисление технических требований в одной строке:
  ```
  TR-1, TR-1.1, TR-1.2, TR-1.3; TR-2, TR-2.1, TR-2.2; ...
  ```

**Созданный документ:**
- Структурированный список по активностям:
  - АКТИВНОСТЬ 1: Войти в приложение (6 функций)
  - АКТИВНОСТЬ 2: Организовать умный дом (9 функций)
  - АКТИВНОСТЬ 3: Управлять умным домом (7 функций)
- Каждая функция указана с названием

**Вывод:** Созданный документ более структурирован и понятен.

---

### 10. Раздел "Примечания"

**Пример:**
- Есть раздел "Примечания" в конце документа:
  - Про использование физического устройства или эмулятора
  - Про соответствие дизайн-системе
  - Про фиксацию результатов скриншотами

**Созданный документ:**
- Раздел "Примечания" отсутствует

**Вывод:** Пример содержит полезные примечания, которые можно добавить.

---

### 11. Количество записей в методике

**Пример:**
- Около 30 записей (TR-1 до TR-7.3)
- Некоторые требования объединены (например, TR-4.4 включает несколько шагов)

**Созданный документ:**
- 35+ записей (все 22 MVP функции + вариации сценариев)
- Каждая вариация сценария (успешный, ошибка) - отдельная запись

**Вывод:** Созданный документ более детальный, покрывает больше сценариев.

---

### 12. Восстановление пароля

**Пример:**
- Использует 4-значный код подтверждения
- Код действует 5 минут
- Процесс: email → код → новый пароль

**Созданный документ:**
- Использует ссылку восстановления
- Ссылка действует 24 часа
- Процесс: email → ссылка → новый пароль

**Вывод:** Разные подходы к восстановлению пароля (код vs ссылка).

---

### 13. Изменение email

**Пример:**
- Использует код подтверждения (4-значный)
- Процесс: новый email → код → подтверждение

**Созданный документ:**
- Использует подтверждение через письмо на новый email
- Процесс: новый email + пароль → письмо на новый email → подтверждение

**Вывод:** Разные подходы к подтверждению смены email.

---

### 14. Добавление устройства вручную

**Пример:**
- Требует выбор типа устройства из справочника
- Упоминает статус и энергосбережение по умолчанию

**Созданный документ:**
- Тип устройства определяется автоматически по коду
- Не упоминает справочник типов

**Вывод:** Разные подходы к добавлению устройства вручную.

---

### 15. Функции, которых нет в примере

**Созданный документ содержит функции, которых нет в примере:**
- US-002: Автосоздание первого дома
- US-044: Удаление устройства из комнаты

**Пример содержит функции, которых нет в созданном документе:**
- TR-7: Выдать доступ к дому (совместный доступ)
- TR-7.1: Принять приглашение

**Вывод:** Разные наборы функций - пример включает совместный доступ, созданный документ фокусируется на MVP без совместного доступа.

## Рекомендации по улучшению созданного документа

1. **Добавить раздел "Примечания"** из примера
2. **Рассмотреть формат основных разделов** - можно сделать более компактным, как в примере (одна таблица)
3. **Уточнить процесс восстановления пароля** - проверить, какой подход используется в вашем проекте (код или ссылка)
4. **Уточнить процесс изменения email** - проверить, какой подход используется (код или письмо)
