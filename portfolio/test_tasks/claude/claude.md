**Статус:** Черновик для согласования  
**Дата:** 2026-04-06  
**Автор:** Аналитик продукта

---

## 1. Краткая формулировка проблемы

### Что решает функция?

AI-оператор — это автоматический агент, который отвечает клиентам в чате вместо живого оператора. Цель: снизить нагрузку на команду поддержки, ускорить время ответа и обрабатывать типовые обращения без участия человека.

### Главная проблема текущего описания

Описание содержит **внутренние противоречия**, **неопределённые термины** и **опасные пробелы в логике**, которые при реализации приведут к конфликтным состояниям системы, ошибкам AI на чувствительных темах и плохому клиентскому опыту.

### Почему нельзя отдавать в разработку в текущем виде

1. **Конфликт управления диалогом** (п. 4 vs п. 5): система одновременно "передаёт диалог оператору" и "продолжает отвечать AI" — непонятно, кто управляет диалогом.
2. **Нет порога уверенности** (п. 3 и 6): "хоть какая-то подходящая информация" — AI будет отвечать неточно или галлюцинировать на чувствительных темах.
3. **Нет таймаута на генерацию** (п. 8): "клиент просто ждёт" противоречит требованию стейкхолдера "клиент не должен долго ждать".
4. **Нет механизма выражения недовольства** (п. 9): неясно, как клиент сигнализирует о недовольстве и что именно триггерит подключение оператора.
5. **Оператор слеп к reasoning AI** (п. 10): без пояснений оператор не может оценить качество ответа и вовремя вмешаться — риск для чувствительных тем.
6. **Race condition** (п. 7): ответ на каждое сообщение по отдельности при быстром вводе приведёт к дублированию и неупорядоченным ответам.

## 2. Требования к системе

Требования сформулированы по принципу: **явные**, **проверяемые**, **однозначные**.

### Функциональные требования

| ID | Требование |
|----|------------|
| FR-01 | Система должна определить **тип обращения** до генерации ответа AI (чувствительная тема / типовой вопрос / нераспознанный запрос). |
| FR-02 | AI должен генерировать ответ только если обращение **не относится к чувствительным темам** (список тем определяется отдельно). |
| FR-03 | AI должен отправлять ответ клиенту только если **уровень уверенности** модели превышает заданный порог (например, ≥ 0.75 — точное значение согласуется с PM). |
| FR-04 | Если AI не может сформировать ответ с достаточным уровнем уверенности, **диалог должен быть передан оператору** с уведомлением (пуш / звуковой сигнал). |
| FR-05 | Если обращение относится к чувствительной теме, **AI не должен отвечать** — диалог немедленно передаётся оператору с пометкой темы. |
| FR-06 | Как только **оператор взял диалог** (подтвердил принятие или отправил сообщение), AI **перестаёт генерировать ответы** по этому диалогу. |
| FR-07 | При **нескольких сообщениях подряд** от клиента (в течение N секунд — значение согласуется) AI должен **дождаться паузы** и отвечать на агрегированный контекст, а не на каждое сообщение по отдельности. |
| FR-08 | Если генерация ответа AI превышает **T секунд** (значение согласуется, например 5–7 с), клиенту отправляется автоматическое сообщение-заглушка: "Мы уже изучаем ваш вопрос, ответим через несколько секунд". |
| FR-09 | Если AI превысил **максимальный таймаут** (например, 15 с), диалог автоматически передаётся оператору. |
| FR-10 | Клиент должен иметь возможность **явно запросить оператора** (кнопка, фраза-триггер). При этом запросе AI прекращает отвечать, диалог уходит к оператору. |
| FR-11 | Оператор видит в интерфейсе **ответ AI + краткое обоснование** (источник из базы знаний / причина передачи). |
| FR-12 | Оператор **не дублирует ответ AI**: если AI уже ответил, оператор видит этот ответ и продолжает диалог с учётом истории. |

### Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-01 | Время ответа AI клиенту — не более **10 секунд** в 95% случаев (P95). |
| NFR-02 | Система должна поддерживать **конкурентную обработку** диалогов без взаимного влияния (изолированные очереди). |
| NFR-03 | Все действия AI (генерация, передача, таймаут) должны **логироваться** для аудита. |
| NFR-04 | Архитектура должна позволять **подключение CRM** как источника данных в следующих этапах без переписывания ядра. |

## 3. Целевая логика работы (исправленная)

### Текстовое описание

1. Клиент отправляет сообщение.
2. Система ожидает паузу N секунд — если клиент продолжает набирать, ждёт окончания (дебаунсинг).
3. Система классифицирует обращение:
   - **Чувствительная тема** → немедленно передать оператору с пометкой, AI не отвечает.
   - **Типовой вопрос / нераспознанный** → AI приступает к генерации ответа.
4. Если генерация занимает > T сек → клиенту отправляется сообщение-заглушка.
5. Если генерация занимает > MAX_TIMEOUT → передать оператору, AI останавливается.
6. AI оценивает уровень уверенности ответа:
   - **Уверенность ≥ порога** → ответ отправляется клиенту, оператор видит ответ + источник в интерфейсе.
   - **Уверенность < порога** → передать оператору с черновиком ответа AI.
7. Если оператор **взял диалог** (принял или ответил) → AI блокируется на этот диалог.
8. Клиент может в любой момент нажать "Позвать оператора" → AI останавливается, диалог передаётся.
9. Оператор видит полную историю: сообщения клиента, ответы AI, источники, причину передачи.

### Схема состояний диалога

```
[Новое сообщение]
       |
       v
[Дебаунс N сек]
       |
       v
[Классификация темы]
    /        \
Чувствит.  Обычная
    |           |
    v           v
[→ Оператор] [AI генерирует]
               |
        ┌──────┴──────┐
      > T сек       ≤ T сек
        |               |
   [Заглушка]      [Оценка уверенности]
        |               /          \
   > MAX_T           ≥ порог     < порог
        |               |            |
   [→ Оператор]   [Ответ клиенту] [→ Оператор
                        |           + черновик]
                   [Оператор видит
                    ответ + источник]
                        |
                  [Клиент доволен?]
                   /           \
                 Да            Нет / кнопка
                  |                |
           [Диалог закрыт]  [→ Оператор]
```

### Таблица: что AI обрабатывает / не обрабатывает

| Тип обращения | AI отвечает? | Комментарий |
|---|---|---|
| FAQ, типовые вопросы по продукту | Да | При уверенности ≥ порога |
| Вопросы по статусу заказа / аккаунту (MVP) | Нет (v2) | Требует CRM-интеграции |
| Медицинские, юридические темы | Нет | Всегда к оператору |
| Жалобы и претензии | Нет | Высокий риск, к оператору |
| Запрос на возврат средств | Нет | Требует авторизации действий |
| Нецензурная лексика / агрессия | Нет | Отдельный сценарий |
| Технические вопросы без ответа в базе | Нет | Низкая уверенность → к оператору |

## 4. Вопросы к Product Manager / заказчику

### Обязательно уточнить до старта

1. **Порог уверенности AI:** какое минимальное значение считается достаточным для отправки ответа? Кто устанавливает и пересматривает?
2. **Список чувствительных тем:** кто ведёт и обновляет этот список? Каков процесс добавления новой темы?
3. **Таймауты:** какие конкретные значения для T (заглушка) и MAX_TIMEOUT (передача оператору)? Есть ли опыт из других продуктов?
4. **Дебаунс:** через сколько секунд паузы считать, что клиент закончил набор? Применять ли ограничение на максимальное число сообщений в очереди?
5. **"Взял диалог" = что именно?** Оператор открыл вкладку, кликнул "Принять", или отправил первое сообщение? Важно для разграничения ответственности.
6. **Механизм недовольства:** как клиент выражает недовольство ответом AI? Кнопка оценки, триггерная фраза, повторный вопрос?
7. **Что видит оператор?** Полный reasoning AI или краткий источник? Как это влияет на производительность оператора?
8. **SLA для оператора:** есть ли максимальное время ответа оператора после получения диалога? Что происходит при превышении?
9. **Многоканальность:** функция только для чата или также email / мессенджеры?
10. **Аналитика и мониторинг:** какие метрики нужно отслеживать с первого дня (% передач оператору, время ответа AI, accuracy)?

## 5. Задача для разработки (черновик)

### Название

**[FEAT] AI-оператор: автоматические ответы в чате с управляемой передачей оператору — MVP**

---

### Проблема / Цель

Операторы перегружены типовыми обращениями. Клиенты ждут ответа дольше допустимого. Необходим автоматический агент, который снимает нагрузку с команды, не допуская ошибок на чувствительных темах и не дублируя работу живых операторов.

---

### Краткое описание решения

Внедрить AI-оператора как первую линию обработки входящих сообщений в чате. AI классифицирует обращение, генерирует ответ при достаточной уверенности и прозрачно передаёт управление оператору в случаях, требующих человека. Оператор видит контекст работы AI и может бесшовно подключиться.

---

### Scope MVP (что входит)

- [ ] Классификатор тем (чувствительная / нечувствительная) на основе заданного списка
- [ ] Генерация ответа AI по базе знаний
- [ ] Оценка уверенности и логика порога
- [ ] Дебаунсинг входящих сообщений (N сек)
- [ ] Заглушка при долгой генерации (> T сек)
- [ ] Автопередача оператору при таймауте и низкой уверенности
- [ ] Блокировка AI при подключении оператора
- [ ] Кнопка "Позвать оператора" для клиента
- [ ] Отображение ответа AI + источника в интерфейсе оператора
- [ ] Базовое логирование действий AI

---

### Out of scope (что не входит в MVP)

- Интеграция с CRM для персонализированных ответов
- Многоканальность (email, мессенджеры)
- Автоматическое закрытие диалога
- Обучение модели на исторических данных
- A/B-тестирование порогов
- Система оценки ответов AI клиентом (рейтинг)
- Аналитический дашборд по метрикам AI

---

### Основные бизнес-правила

1. AI **не отвечает** на чувствительные темы — только оператор.
2. AI отправляет ответ клиенту **только при уверенности ≥ CONFIDENCE_THRESHOLD**.
3. Если оператор **взял диалог** — AI **не генерирует** новые ответы в этом диалоге.
4. Клиент всегда может **запросить оператора** вручную — AI останавливается.
5. Оператор **видит все действия AI** (ответ, источник, причину передачи) для оценки качества.

---

### Требования к системе

> Ссылка на полный перечень: Раздел 2 данного документа (FR-01 — FR-12, NFR-01 — NFR-04)

Ключевые параметры для реализации (значения согласуются с PM):

| Параметр | Название | Значение по умолчанию |
|---|---|---|
| `DEBOUNCE_TIMEOUT` | Пауза перед обработкой сообщений | 2–3 сек |
| `CONFIDENCE_THRESHOLD` | Минимальная уверенность AI | 0.75 |
| `TYPING_INDICATOR_TIMEOUT` | Таймаут до заглушки | 5 сек |
| `MAX_GENERATION_TIMEOUT` | Максимальное время генерации | 15 сек |

---

### Критерии приёмки

- [ ] AI не отвечает на обращения из списка чувствительных тем; диалог уходит к оператору с пометкой темы.
- [ ] AI не отправляет ответ при уверенности < CONFIDENCE_THRESHOLD; диалог уходит к оператору с черновиком.
- [ ] При получении сообщений с паузой < DEBOUNCE_TIMEOUT система ждёт и обрабатывает агрегированный контекст.
- [ ] При генерации > TYPING_INDICATOR_TIMEOUT клиент получает сообщение-заглушку.
- [ ] При генерации > MAX_GENERATION_TIMEOUT диалог автоматически передаётся оператору.
- [ ] После того как оператор взял диалог, AI не генерирует новых ответов в этом диалоге.
- [ ] Нажатие кнопки "Позвать оператора" останавливает AI и передаёт диалог.
- [ ] Оператор видит ответ AI и источник (или причину передачи) в интерфейсе.
- [ ] Все действия AI записываются в лог с меткой времени и диалога.
- [ ] P95 времени ответа AI клиенту ≤ 10 сек.

## Приложение: Предложение по этапам

| Этап | Содержание | Цель |
|---|---|---|
| **MVP** | Классификатор тем, генерация по базе знаний, порог уверенности, передача оператору, блокировка AI | Снизить нагрузку на 20–30%, не допустить ошибок на чувствительных темах |
| **v1.1** | Кнопка оценки ответа AI клиентом, базовый аналитический дашборд, настройка порогов через UI | Измерить качество, дать операторам инструмент обратной связи |
| **v2.0** | Интеграция с CRM, персонализированные ответы, A/B-тесты порогов | Повысить автоматизацию до 50–60%, уменьшить время ответа |
| **v3.0** | Дообучение модели на исторических диалогах, мультиканальность | Переход к умному оператору с памятью контекста клиента |
